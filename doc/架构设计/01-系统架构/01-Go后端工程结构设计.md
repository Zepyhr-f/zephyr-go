# Go 后端工程结构设计 (基于 go-zero)

> 本文档规定 Zephyr Admin (Go重构版) 的 Monorepo 代码仓库目录结构、各服务（微服务/网关）职责边界、代码分层约定，是所有后端开发者的**行为准则**。

---

## 1. 总体架构分层

系统采用 **BFF 网关 + 内部 RPC 微服务** 的架构，全部代码收敛于同一个 Monorepo 仓库。

```text
全局请求入口 (Nginx / Load Balancer)
      ↓
API BFF 网关 (app/gateway)  【提供统一对外 HTTP 接口，负责统一认证(JWT)、粗粒度鉴权、参数校验】
      ↓
内部 RPC 契约 (gRPC / Protobuf)
      ↓
后端核心微服务集群
  ├── 统一认证域 (app/auth) 【提供多策略登录、双 Token 签发验证、扫码/验证码扩展】
  ├── 身份与租户域 (app/identity, app/tenant) 【提供多租户体系、用户基础数据与隔离权限控制】
  ├── AI 核心能力域 (app/rag, app/agent) 【提供大模型代理、提示词调度、知识库检索增强】
  └── 工作流编排域 (app/workflow) 【提供 Agent 任务与外部工具的 DAG 工作流编排引擎】
      ↓
数据访问层 (pkg/orm -> MySQL / Redis)
```

## 2. Go Monorepo 目录结构规划

代码仓库采用统一的 `go.mod` 文件管理所有依赖，按应用（`app`）和公共包（`pkg`）划分边界：

```text
zephyr-go/
├── go.mod                            # 根级的依赖管理文件，统一管控所有依赖版本
├── app/                              # 应用层：包含所有的网关与微服务
│   ├── gateway/                      # ①【网关层】BFF API 聚合网关 (HTTP入口、统一鉴权、反向代理)
│   │   ├── etc/                      # gateway.yaml (配置文件)
│   │   ├── internal/                 # goctl 生成的 controller/logic 层
│   │   └── gateway.api               # HTTP API 接口定义 DSL (主入口)
│   │
│   ├── auth/                         # ②【统一认证域】多策略登录(账号/短信/扫码)、双Token签发、登出 (gRPC)
│   ├── identity/                     # ③【身份体系域】用户资料CRUD、角色权限分配、菜单树维护 (gRPC)
│   ├── tenant/                       # ④【多租户域】租户管理、资源隔离配置、组织架构 (gRPC)
│   ├── rag/                          # ⑤【检索增强域】知识库构建、文档解析切片、向量检索 (gRPC)
│   ├── workflow/                     # ⑥【工作流域】DAG 任务编排、流程状态流转引擎 (gRPC)
│   └── agent/                        # ⑦【AI 代理域】大模型接入、Agent 能力定义 (以其为例展开内部结构)
│       ├── etc/                      # agent.yaml
│       ├── agent.proto               # gRPC 接口定义 (直接在微服务根目录，拒绝过深嵌套)
│       ├── pb/                       # 根据 proto 自动生成的存根代码
│       └── internal/
│           ├── logic/        # 【DDD 应用层】用例编排 (复用 goctl 生成)
│           ├── domain/       # 【DDD 领域层】Agent 实体、Prompt 值对象、领域服务
│           ├── repository/   # 【数据层】数据库持久化实现 (GORM DAO)
│           ├── adapter/      # 【适配层】外部大模型 API 适配器、RPC 防腐层
│           └── server/       # 【DDD 接口层】gRPC 接入端 (复用 goctl 生成)
│
├── pkg/                              # ④【公共库】原 zephyr-framework 的 Go 语言平替
│   ├── core/                         # 核心工具：统一响应结构 (R)、错误码定义、常用 Utils
│   ├── orm/                          # ORM 封装：基于 GORM 的通用初始化、分页封装、自动审计字段填充
│   └── jwtx/                         # 认证处理：JWT 解析验证、双令牌逻辑封装
│
└── scripts/                          # 部署与开发脚手架
    ├── docker-compose.yml            # 本地依赖启动 (MySQL, Redis, ETCD)
    └── gen.sh                        # 用于执行 goctl 生成代码的快捷脚本
```

## 3. 各模块职责详解

### 3.1 `pkg/core` — 核心工具与响应标准

**职责**：提供全项目通用的底层数据结构，绝不携带具体业务逻辑。

- **`response.go`**：统一封装 HTTP 响应。定义结构类似 Java 的 `R<T>`：`{ "code": 200, "msg": "success", "data": {...} }`。
- **`xerr.go`**：定义统一业务错误码，并支持结合 i18n 转化为可读提示。
- **`utils/`**：常用字符串处理、加密（Bcrypt）、时间转换等。

---

### 3.2 `pkg/orm` — 持久层（基于 GORM）

**职责**：封装数据库连接管理，并建立标准化实体。

- **`base_model.go`**：定义标准表结构，平替 Java 的 `BaseEntity`，包含：
  - `ID` (主键)
  - `CreatedAt`, `UpdatedAt` (由 GORM 自动维护)
  - `CreatedBy`, `UpdatedBy` (从上下文中提取的用户 ID)
  - `DeletedAt` (利用 GORM 自带的软删除 `gorm.DeletedAt`)
- **自动填充**：通过 GORM 的 Hooks（如 `BeforeCreate`, `BeforeUpdate`）读取上下文中透传的用户标识实现创建人/更新人的自动填充。

---

### 3.3 `app/gateway` — BFF 聚合网关

**职责**：作为前端所有请求的唯一入口，承担鉴权、路由分发与数据聚合。

- **接口定义**：在 `gateway.api` 中描述所有的 HTTP 接口路径和对应的 Request/Response DTO 结构。
- **JWT 鉴权拦截**：在 API 配置中开启 `jwt` 配置项，由 go-zero 原生支持 Token 校验。
- **角色/权限校验**：在 HTTP Middleware 中拦截请求，根据 Token 中的角色或 UserID 拉取 Redis 缓存校验接口访问权限（URL-RBAC）。
- **请求分发**：Gateway 通过配置注册中心（如 ETCD），调用后端的 `identity`, `tenant`, `rag`, `agent`, `workflow` 等模块的 RPC 接口获取数据并聚合返回。

#### 🌟 进阶技巧：`gateway.api` 文件的拆分管理建议

规范将 `gateway.api` 直接放在了网关根目录。随着系统演进（包含 6 大核心业务域），这个 API 文件在重构完后可能会膨胀至数千行，极其难以维护。

**建议方案**：利用 `go-zero` 的 `import` 语法，按业务域将 API 描述文件拆分，最后在主文件中聚合。

```text
app/gateway/
├── gateway.api            # 主入口，import 拆分后的子文件
└── desc/
    ├── identity.api       # 身份域接口定义
    ├── tenant.api         # 租户域接口定义
    ├── agent.api          # 代理域接口定义
    └── ...
```

在主文件 `gateway.api` 中通过如下语法聚合：
```go
import "desc/identity.api"
import "desc/tenant.api"
import "desc/agent.api"
```
采用这种方式，既避免了顶层配置文件的过深嵌套，又优雅地解决了单文件臃肿的问题，使得各领域接口独立迭代互不干扰。

### 3.4 结合 DDD 的微服务分层规范 (`internal/`)

为了解决传统 `go-zero` 默认代码生成容易演变为“贫血模型（Transaction Script）”的问题，我们在微服务内部（尤其是核心 AI 业务域如 `app/agent`, `app/workflow`）引入 **DDD（领域驱动设计）** 的四层架构约束。

我们将 `goctl` 原生的目录生成机制与 DDD 分层进行无缝融合，具体结构如下：

```text
internal/
├── server/             # 【接口层 Interfaces】由 goctl 自动生成的 RPC Server 绑定，负责协议转换。
├── handler/            # 【接口层 Interfaces】(针对 API 服务) HTTP 路由绑定与参数解析。
├── logic/              # 【应用层 Application】由 goctl 生成。作为 Application Service，负责用例编排、事务控制，不包含纯业务规则。
├── domain/             # 【领域层 Domain】(需手动创建) 系统的核心，无任何外部依赖：
│   ├── entity/         # 领域实体 (包含核心业务属性与行为，富领域模型)
│   ├── valobj/         # 值对象 (Value Object)
│   ├── service/        # 领域服务 (处理跨多个实体的复杂领域逻辑)
│   └── irepository/    # 仓储接口约束 (定义数据持久化契约，也可直接在 repository 中定义)
├── repository/         # 【数据层 Repository】(需手动创建) 数据库持久化实现：
│   └── agent_repo.go   # 仓储实现 (内部封装 GORM 和底层表模型 BaseEntity，实现对 MySQL 的操作)
├── adapter/            # 【适配层 Adapter】(需手动创建) 外部服务防腐层：
│   └── llm_client.go   # 包装对其他服务 RPC Client 或大模型第三方 API 的调用，统一错误处理
├── svc/                # 服务上下文，用于依赖注入（将 repository/adapter 的实例注入给 logic 层）
├── config/             # 对应 .yaml 映射的结构体
└── types/              # (仅 API) 请求和响应的类型定义
```

**DDD 依赖控制原则**：
1. **防腐与倒置**：`logic` 层依赖接口约束，绝不能直接写 SQL 或引入 GORM。真正的 MySQL/GORM 操作必须下沉收拢到 `repository` 目录中，外部调用收拢在 `adapter` 目录中，通过 `svc` (ServiceContext) 依赖注入，将核心业务与具体技术实现解耦。
2. **富模型驱动**：将原本堆砌在 `logic` 中的判断和计算，下沉并封装到 `domain/entity` 的方法中，保证业务逻辑的内聚。

## 4. API 契约与自动生成

在 Go 工程中，我们以 **Schema-first (契约优先)** 驱动开发。

1. **HTTP 接口开发**：
   开发者先编写 `app/gateway/gateway.api` 文件。
   在 `app/gateway` 目录下执行指令：`goctl api go -api gateway.api -dir .`
   框架自动生成 HTTP 路由、参数解析代码、以及空的 Logic 方法，开发者只需在 Logic 中填入调用后端 RPC 的代码。

2. **gRPC 接口开发**：
   开发者编写 `app/identity/identity.proto` 文件。
   在 `app/identity` 目录下执行指令：`goctl rpc protoc identity.proto --go_out=./pb --go-grpc_out=./pb --zrpc_out=.`
   框架自动生成 Protobuf 结构、gRPC 服务端骨架以及对外的 Client 代码。

## 5. 迁移落地路线图

| 阶段 | 内容 | 行动 |
|---|---|---|
| **P1（基建建设）** | `zephyr-go` Monorepo 骨架搭建，配置统一 `go.mod` 与 `docker-compose.yml` (含 ETCD, Redis, MySQL, 向量数据库) | 初始化 `pkg/core`, `pkg/orm` |
| **P2（租户与身份底座）** | 构建 `identity` 和 `tenant` 服务，落地多租户隔离与 RBAC 权限体系，并在 `pkg/orm` 验证 GORM | 使用 `goctl` 建立基础 RPC 服务 |
| **P3（网关赋能）** | 构建 `gateway-api`，开发 `/auth/login` 及 JWT 双令牌签发验证功能，代理对基座服务的访问 | 实现 JWT Middleware 和路由转发 |
| **P4（AI 核心引擎建设）** | 依照 DDD 规范，逐步构建 `rag` (知识库构建与向量检索) 与 `agent` (大模型交互与提示词调度) 核心引擎 | 构建大模型适配与富领域模型 |
| **P5（工作流编排与上线）** | 构建 `workflow` 串联各项能力，完成 CI/CD 流程切换，通过 Docker 构建网关及各微服务镜像并联合部署 | 完善部署脚本 |
