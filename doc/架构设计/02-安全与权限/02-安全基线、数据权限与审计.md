# 安全基线、数据权限与审计

> 本文档整合安全与权限的横切规则：安全基线（工程约束）+ 数据权限（DataScope）+ 审计（Audit）。  
> 登录/鉴权链路、Token 与网关验签细节请参见《[登录与鉴权全链路](./01-登录与鉴权全链路.md)》。

---

## 1. 安全基线（必须遵守）

### 1.1 密钥与配置安全
- 禁止将密钥（JWT 私钥、数据库密码、AK/SK）提交到 Git
- 密钥统一通过配置中心/环境变量注入，按环境隔离（dev/test/prod）
- 日志中禁止打印敏感字段（密码、token、身份证、手机号全量等）

### 1.2 身份认证与会话
- Access Token 短期有效（建议 15~30min），Refresh Token 长期有效（建议 7d）
- Refresh Token 必须可吊销（Redis 存储 + 退出时删除）
- Token 黑名单：对 Access Token 的 jti 做失效记录（TTL=剩余有效期）

### 1.3 密码与账号安全
- 密码存储必须使用强哈希（BCrypt/Argon2），禁止明文/可逆加密
- 登录失败次数限制、验证码/二次验证策略（按风险等级启用）
- 后台账号强制复杂度与定期更新策略（按产品要求）

### 1.4 API 安全
- 参数校验：所有写接口必须 `@Valid`，避免脏数据入库
- 防重放：网关签名/时间戳策略（启用即强制）
- 限流：网关或服务端限流（按接口等级）
- CORS：仅允许可信域名（生产环境严禁 `*`）

### 1.5 数据安全与隐私
- 脱敏：手机号/身份证/邮箱等按展示场景脱敏
- 数据权限强制：所有列表/导出接口必须经过 DataScope
- 导出与下载：权限 + 审计 + 频控（避免批量泄露）

## 2. 数据权限（DataScope）

### 2.1 原则
- **服务侧强制**：数据权限必须在业务服务内部落地，网关只做粗粒度鉴权
- **默认最小权限**：无明确授权默认不可见/不可操作
- **可解释**：权限过滤规则应可定位、可追溯（便于排障）

### 2.2 常见数据范围
- 仅本人（SELF）
- 本部门（DEPT）
- 本部门及子部门（DEPT_AND_CHILD）
- 全部（ALL）
- 自定义（CUSTOM：按部门/组织/岗位集合）

### 2.3 推荐实现方式
（结合 MyBatis-Plus）
- Controller/Service 层确定 DataScope（从 Token/权限系统得到）
- 在查询条件 DTO 中携带“范围参数”（不要让前端直接传“范围”）
- Mapper 层通过动态 SQL 拼接范围条件，或通过统一拦截器注入（如后续需要）

## 3. 审计（Audit）

### 3.1 审计范围
至少覆盖：
- 登录/登出、刷新 Token、修改密码
- 权限变更：用户角色、菜单权限、数据范围变更
- 关键业务写操作：创建/修改/删除（按业务域定义）

### 3.2 审计内容（建议字段）
- 操作人：userCode、roles
- 租户/组织：tenantCode（如有）
- 操作对象：resourceType、resourceId
- 操作类型：CREATE/UPDATE/DELETE/LOGIN...
- 请求信息：path、method、ip、userAgent、traceId
- 结果：success、errorCode、errorMsg
- 耗时：durationMs

### 3.3 落地位置（推荐）
- 网关：生成/透传 `traceId`，注入基础 Header（用户信息、租户等）
- 服务侧（平台域/业务域）：在 Service 层采集审计事件
- 审计写入：可同步落库（简单可靠）或异步 MQ（高吞吐、需幂等）
