## 1. 整体链路与职责边界

```
前端请求
  ↓  Authorization: Bearer {accessToken}
Gateway（认证 + URL-RBAC 粗粒度鉴权 + 注入可信 Header + X-Gateway-Sign）
  ↓  可信 Header（X-UserCode / X-TenantCode / X-Roles / X-Request-Id / X-Gateway-* / X-Gateway-Sign）
后端服务（验签 + 细粒度权限/perms + 数据权限/DataScope + 业务处理）
  ↓
MySQL / Redis / 其它依赖
```

强制约定：
- **网关负责**：认证（JWT + 黑名单）+ URL-RBAC（粗粒度）+ 可信 Header 注入 + 网关签名
- **服务负责**：验签后信任 Header；执行细粒度权限（按钮级 perms）与数据权限（DataScope）

## 2. Token 方案（摘要）

> Access Token = JWT（短）；Refresh Token = Cookie（长，可吊销）

### 2.1 Access Token（JWT）
- 载体：`Authorization: Bearer {token}`
- 建议 payload：`userCode`、`tenantCode`、`iat`、`exp`、`jti`
- **JWT 不承载 roles/perms**：roles/perms 以 Redis（或服务侧查询）为准，网关可做短 TTL 本地缓存降压

### 2.2 Refresh Token
- 载体：HttpOnly Cookie（建议 `Secure=true`、`SameSite=Strict`、Path 最小化）
- 服务端可吊销：Redis 存储 refresh 映射或 jti（退出/改密时删除）

### 2.3 Redis Key（关键约定）
```
refresh:{jti} -> userCode + tenantCode (TTL=7d)
blacklist:{jti} -> 1 (TTL=access 剩余有效期)

auth:user:roles:{tenantCode}:{userCode} -> Set<String> roleCodes
auth:path:roles:{tenantCode}:{httpMethod}:{pathPattern} -> Set<String> roleCodes
```

## 3. 前端对接约定

### 3.1 请求头规范
```
Authorization: Bearer {accessToken}
X-Request-Id: uuid
X-Client-Type: web
```

### 3.2 刷新流程（必须加锁）
```
1) 任一请求返回 401
2) 调用 /auth/refresh（自动携带 Refresh Cookie）
3) 获取新 Access Token
4) 重放原请求
```

### 3.3 登录后初始化（两步）
1) `GET /system/user/info`：用户信息 + roles + permissions（按钮级 perms）  
2) `GET /system/menu/routers`：动态菜单与路由树（前端只挂载后端返回的路由）

## 4. 网关鉴权（摘要）

### 4.1 路径分类
1) 白名单：`/auth/login`、`/auth/refresh` 等（不校验 token）  
2) 基础认证：校验 token，但不做 URL-RBAC（如 `/system/user/info`）  
3) 严格鉴权：校验 token + URL-RBAC（角色-路径）

### 4.2 透传 Header（必须统一）
```
X-UserCode
X-TenantCode
X-Roles
X-Request-Id
X-Gateway-Timestamp
X-Gateway-Nonce
X-Gateway-Sign
```

### 4.3 网关签名（核心）
```
canonicalString =
  timestamp + "\n" +
  nonce + "\n" +
  method + "\n" +
  path + "\n" +
  tenantCode + "\n" +
  userCode + "\n" +
  requestId

sign = base64(HMAC_SHA256(gateway-secret, canonicalString))
```

## 5. 后端服务验签与用户上下文

### 5.1 用户上下文模型（建议）
```java
public class UserContext {
    private String userCode;
    private String tenantCode;
    private java.util.Set<String> roles;
    private String requestId;
    private String traceId;
}
```

### 5.2 处理流程（建议）
```
请求进入
  ↓
拦截器解析 Header（userCode/tenantCode/roles/requestId）
  ↓
校验 X-Gateway-Sign + timestamp（防伪造/防重放）
  ↓
写入“请求级上下文”（生命周期随请求结束清理）
  ↓
业务处理（细粒度权限 + DataScope）
```

---

## 6. 跨服务调用（Feign）约定

### 6.1 必须透传的 Header
同网关透传 Header（见 4.2），由 Feign `RequestInterceptor` 自动注入。

### 6.2 注意事项
- 下游服务**同样必须验签**，否则 Header 可被伪造
- 跨服务调用的超时/重试策略要显式配置，避免故障放大
